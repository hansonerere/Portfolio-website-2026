# 🔄 网站更新工作流程

## 🎯 自动同步原理

### 现在的设置：
```
本地代码修改 → Git推送到GitHub → Netlify自动部署 → 网站更新
                                ↘ Vercel自动部署  ↗
```

**两个平台都会同时自动更新！**

## 📝 标准更新流程

### 第一步：修改代码
在本地项目中修改任何文件：
- 修改组件文件 (如 `App.tsx`)
- 更新样式 (如 `styles/globals.css`)  
- 添加新功能
- 修复bug

### 第二步：提交更改到Git
```bash
# 进入项目目录
cd /Users/hanwu/Desktop/hansonpage

# 查看修改的文件
git status

# 添加所有更改
git add .

# 创建提交（描述你的更改）
git commit -m "更新：描述你做了什么修改"

# 推送到GitHub
git push
```

### 第三步：等待自动部署
- **Netlify** 会自动检测到更改并开始构建
- **Vercel** 也会同时自动构建
- 通常 **2-5分钟** 后网站就更新了！

## 📊 部署状态监控

### Netlify监控：
1. 登录 [netlify.com](https://netlify.com)
2. 进入您的站点
3. 查看 "Deploys" 标签页
4. 看到构建状态：Building → Success

### Vercel监控：
1. 登录 [vercel.com](https://vercel.com)  
2. 进入您的项目
3. 查看部署历史
4. 确认构建成功

## 🔄 不同类型的更新

### 1. 代码更新（需要重新部署）
- 修改React组件
- 更新样式文件
- 添加新功能
- 配置文件更改

**流程**：修改 → Git推送 → 自动部署

### 2. 内容更新（无需重新部署）
- 修改Supabase数据库内容
- 更换图片URL
- 更新文字内容

**流程**：直接在Supabase修改 → 立即生效

## 🚨 注意事项

### ✅ 推送前检查：
- 确保代码能正常运行
- 检查是否有语法错误
- 测试新功能是否正常

### ⚠️ 如果部署失败：
1. 查看Netlify/Vercel的构建日志
2. 检查错误信息
3. 修复问题后重新推送

### 🎯 最佳实践：
- 每次提交写清楚的说明
- 大功能分多次小提交
- 重要更新前先在本地测试
